# Topic Architecture Planner

## Overview

Use this skill to decide what kinds of pages a topic needs, what each page should be responsible
for, and how they should connect. Focus on page roles, search-intent separation, collection-page
design, and clean hub-to-supporting-page relationships.

## Workflow

### 1. Identify the topic and intents

Before drawing structure, separate the topic into intents such as:

- definition or overview
- process or how-to
- pricing or cost
- comparison
- FAQ
- local or service-area variants
- special-case pages

If multiple pages currently target the same intent, flag cannibalization before proposing new
pages.

### 2. Assign page roles

For each topic, define which page should be:

- hub page: the main collection or overview page
- supporting page: deeper subtopic or single-intent page
- conversion page: service or contact-oriented page
- utility page: FAQ, glossary, checklist, document list

Do not let every page try to be the hub.

### 3. Decide what belongs on the hub page

A good collection page is not just a list of posts. It should usually include:

- a short overview of the topic
- decision or scenario-based entry points
- grouped links to the most important subpages
- a concise summary of core facts
- a CTA or next step

Treat hub pages as navigation and synthesis pages, not archive dumps.

### 4. Keep search intents separated

If the topic has distinct intents, split them into separate pages instead of merging everything.

Examples:

- `拋棄繼承是什麼` -> overview page
- `拋棄繼承流程` -> process page
- `拋棄繼承期限` -> deadline page
- `拋棄繼承費用` -> pricing page
- `未成年人拋棄繼承` -> special-case page

The hub page should summarize and route, not fully replace each child page.

### 5. Plan the internal-link structure

For each cluster, define:

- hub -> child links
- child -> hub link
- child -> sibling links where helpful
- CTA paths from informational pages to service or contact pages

Prefer simple, obvious paths. Keep important pages within a shallow click depth.

### 6. Recommend layout patterns by page type

For hub or collection pages, prefer this order:

1. hero with topic framing
2. quick decision paths or grouped navigation
3. concise explanatory summary
4. key supporting-page links
5. FAQ or trust elements
6. CTA

For supporting pages, prefer:

1. direct answer to the intent
2. scoped detail
3. links back to hub and sideways to relevant siblings
4. CTA when appropriate

### 7. Flag weak architecture patterns

Call out these problems explicitly:

- one page trying to rank for every intent
- collection pages with no original synthesis
- multiple pages with the same keyword target
- supporting pages with no link back to the hub
- service pages competing with informational pages

### 8. Produce a page-role plan before editing

Before proposing copy or design changes, summarize:

- page inventory
- intended role for each page
- which page should rank for which intent
- what new pages are needed
- what pages should merge, demote, or stop competing

When useful, use a table like:

| Page | Role | Primary intent | Keep / change / merge | Notes |
|------|------|----------------|------------------------|-------|

## Do Not Do These

- Do not make every page a hub
- Do not merge distinct search intents into one bloated page
- Do not leave collection pages as bare post lists
- Do not propose new pages without first mapping intent conflicts
- Do not ignore the internal-link paths that support the architecture
